<h2>4.1. Удачные модели ветвления в Git</h2>
Разные команды разрабатывают и используют свои методологии при работе с git/ветвлением в git.

На данный момент есть две популярные модели ветвления в Git:

- **git flow**
- **github flow**

Они имеют как много общего, так и много различий. 
Стали они популярны благодаря тому, что абстрагируются от отраслей разработки. 

<h2>Git flow</h2>
В этой модели есть набор правил, по которым можно организовывать ветки в своём репозитории.
Есть несколько видов веток:
- Две главные ветки(master, develop);
- Ветки, в которых разрабатывается новая функциональность(feature);
- Ветки, в которых делаются быстрые фиксы(hotfix);
- Ветка в которой формируем новую версию программного продукта(release).

<h3>Ветки master и develop</h3>
Ветка master считается главной в большинстве моделей ветвления git, потому что создаётся по-умолчанию, при создании репозитория.
Ветка develop создаётся дополнительно в рамках использования модели gitflow.

В ветке master всегда должен быть код, готовый к исполнению. Применительно к использованию сайта, код в ветке master должен быть всегда готов к отправке на сервера. 
В ветке develop в базовом представлении находится код из ветки master + новые функции.

Gitflow предполагает, что в ветки master и develop напрямую коммитить нельзя. В этих ветках содержится результат работы, которая была произведена в других ветка репозитория. 

<h3>Ветки feature</h3>
В этой ветке разрабатывается новая функциональность или делаются большие изменения с уже имеющейся функциональностью. 
*Создаётся из ветки develop*. В ней происходит работа над новой функциональностью. Когда функциональность готова - ветка feature смерживается с веткой develop.

![[Pasted image 20260429103224.png]]

Также модель gitflow предполагает собственное именование веток. При создании ветки feature пользуемся следующим правилом: **feature/название_которое_мы_дали**.
Самыми долгоживущими ветками после master и develop являются ветки feature, потому что менять текущую функциональность или разрабатывать новую можно очень долго. 

<h3>Ветки hotfix</h3>

Используются для того, чтобы делать быстрые исправления багов в проекте. 

По gitflow ветки hotfix создаются из ветки master. После исправления критичных ошибок в ветке hotfix она мержится с ветками main и develop (для того, чтобы исправления из hotfix было в обоих версиях приложения).

<h3>Ветки release</h3>
Нужны для того, чтобы поставлять новую функциональность в продакшн (в ветку master). 
Часто в ветках release делается увеличение версии ПО. 

![[Pasted image 20260429104532.png]]

После работы в ветке release она мержится с ветками master и develop, а также создаётся git tag (сущность, с помощью которой можно отслеживать какие релизы были созданы в репозитории). 

Это основной набор веток, который используется при работе с моделью gitflow. Но существуют также дополнения к этому набору в виде веток patch.

Есть одноимённая утилита, которая облегчает использование модели gitflow, которая автоматизирует и организовывает работу в соответствии с правилами модели. 

**git flow init** - команда для инициализации утилиты.

**git flow feature start newFeature** - создание ветки feature/newFeature.

**git flow feature finish newFeature** - завершение работы с веткой feature/newFeature.

**git flow release start 1.0.1** - создание ветки release 1.0.1. Название ветки должно быть равно номеру релиза, который мы делаем. В данном случае 1.0.1.

<h2>Github flow</h2>

У этой модели есть сходство с gitflow - наличие общей ветки. Но в отличие от gitflow, где таких ветки две, в github flow - одна. Обычно это ветка master (иногда её называют main).

Эта модель проще, потому что мы имеем ветку main, из которой в любой момент мы можем отправлять код в продакшн. И от ветки main создаются дополнительные ветки с названиями говорящими о том, что в этих ветках делаются. 

Особого соглашения по именованию веток нет. 

<h2>4.2. Code-review и как Gitlab помогает его организовать.</h2>
<h3>Функции хостингов разработки ПО 
Pull Request(Github) и Merge Request(Gitlab)</h3>
Эти функции помогают мержить ветки, параллельно делая много дополнительных действий. 
Самый частый процесс, для которого используются эти функции - code-review.

**Code-review (от англ. - просмотр кода)** - процесс при котором один разработчик просматривает код другого разработчика при попытке смержиться в основную ветку. 

<h2> 4.3. Что такое CI / CD? Как с ними интегрируется Git?</h2>

**CI - Continuous integration** - системы непрерывной интеграции. Позволяют специалистам автоматически запускать процессы, которые раньше запускались вручную. 

Например: 
- компиляция;
- прогон автоматических тестов.

**CD - Continuous delivery**

Если обобщить и назвать по-русски - **системы непрерывной интеграции и непрерывной доставки.**

Большая часть настроек системы непрерывной интеграции происходит в репозитории в конкретном файле, который в случае с GitLab называется **.gitlub-ci.yml**.

В файле происходит **декларативное** описание действий, которое предполагает, что программист описывает не то, как сделать определённое действие, а какой результат он ждёт в конце. 

Чтобы упростить связь между всеми членами команды разработчиков и исполняемыми процессами существуют системы непрерывной интеграции. Они позволяют автоматически запускать процессы, которые программисту пришлось бы делать вручную, ликвидируют возможные ошибки, ускоряют разработку любого программного продукта. 

В случае с CD мы имеем не только плюсы автоматического запуска процессов, а ещё связь с конечными серверами и другими сущностями, чтобы доставить обновления до пользователей. 

В современном мире наличие CI / CD в команде разработки является одним из критериев определения хорошая команда или плохая. 

```
run_on_develop_branch:   
	script:       
		- echo “SUCCESS RUN ONLY ON develop”   
	only:       
		refs:           
			- develop
```

**run_on_develop_branch** - название задачи
**script** - описывает то, что будет дальше происходить в задаче run_on_develop_branch.
**echo ""SUCCESS RUN ONLY ON develop"** - команда на языке bash для отображения сообщения. 
**only** - указание на запуск задачи в определённом месте. 
**refs** (сокращение от references) - технически ссылки на коммиты. В примере ссылаемся на ветку develop.

Исходя из конфигурационного файла, задача будет запускаться по каждому коммиту в ветке develop.
