<h2> 3.2. Что такое репозитории и как ими пользоваться? </h2>
**Репозиторий** - одна из основных сущностей при работе с Git. В общем представлении - хранилище данных.

**git init** - превращение текущей папки в git-репозиторий. 
**git clone** - копирует удалённый репозиторий в локальный. 

<h2>3.3. Коммиты. Как сохранять изменения в репозитории?</h2>
**Коммит** (от англ. commit - сохранение, фиксация). Фиксация изменений файлов и данных в репозитории.
Коммиты могут содержать информацию о:

- изменении в файлах;
- добавлении новых файлов;
- удалении файлов.

**git add . -p** - добавление изменений в индекс. (-p или --patch - просмотр изменений в репозитории в интерактивном формате)
**git commit -m "Second commit"** - более краткая версия команды с указанием комментария к коммиту в команде. 

**git commit --allow-empty -m "empty commit"** 
Иногда при решении проблем бывает полезно сделать пустой коммит. Это коммит, в котором нет данных об изменении/удалении/добавлении файлов. 

**git log** - показывает список коммитов. 

**git rm 1.txt** - удалить файл с добавлением в индекс коммита

**git checkout 1.txt** - вернуть файл в состояние предыдущего коммита, отменить последние изменения в файле. 

**git checkout .** - вернуть все файлы в текущей директории в состояние предыдущего коммита.

<h2>3.4. Ветвление в Git</h2>
**Ветки** - указатели на коммиты (официальная документация). 

**git branch** - показывает список доступных веток в локальном репозитории и в какой ветке находимся на данный момент. 

**git branch newBranch** - создание новой ветки от последнего коммита.

**git checkout -b newBranch** - создание новой ветки и переключение на неё.

*(дана инструкция по установке скрипта, который отображает текущую ветку в терминале при приглашении к вводу, может быть полезно)*

**git branch -D develop** - удалить ветку develop. 

**git branch -m slurm_develop** - переименовать текущую в slurm_develop.

**git checkout HEAD^** - вернуться на один коммит назад относительно текущего.

<h2>3.5. Обмен изменениями с удалёнными репозиториями</h2>
**git push** - команда для отправки изменений в текущей ветке в удалённый репозиторий соответствующей ветки

**git push --force**  - принудительно перезаписывает историю коммитов в ветке, в которой команда была вызвана. Лучше не использовать! Вместо гарантии сохранения данных и решения конфликтов данные стираются!

**git push -u gitlab_repo master** - создаём ветку master в нашем удалённом репозитории gitlab_repo (-u = --set-upstream).

**git pull** (от англ. - тащить, вытащить) - команда без параметров забирает в локальный репозиторий изменения из всех веток, которые были сделаны в удалённом репозитории. По умолчанию обращается к основному репозиторию origin. 

**git pull gitlab_repo** - забрать изменения из удалённого репозитория gitlab_repo.

**git remote** - проверить с какими удалёнными репозиториями связан наш локальный репозиторий в данный момент.

**git remote remove gitlab_repo** - удалить связь локального репозитория с удалённым gitlab_repo.

**Тэги в Git** - записи, которыми можно отмечать конкретные версии продукта. 

**git tag** - отображение тэгов в репозитории.  Если тэгов нет, то команда ничего не вернёт. 

**git tag -a 0.1** - флаг (-a) отвечает за создание тэгов. Команда создаёт тэг 0.1.

**git push --tags** - флаг (--tags) говорит о том, что отправляем в удалённый репозиторий именно тэги. 

**git fetch** - скачивает изменения из удалённого репозитория, обновляя данные, но не меняя рабочие файлы или текущую ветку.

**git branch -r** - просмотр веток удалённого репозитория.

**git fetch origin develop:myOwnDevelop** - копирование ветки develop из удалённого репозитория в локальный с изменением имени на myOwnDevelop.

<h2>3.6. Сложные кейсы слияния веток</h2>
**git merge** - команда для слияния веток

У каждого коммита есть родительский коммит за исключением первого коммита. 
Ветка А - это не хранилище всех коммитов в ветке A, а просто ссылка на последний коммит в ветке A. Дальнейшее представление ветки складывается по ссылкам коммита на родительский коммит.

После **git merge** создаётся *merge-commit*, у которого два родителя. Первый - последний коммит из ветки A, второй - последний коммит из ветки B.  Все изменения из ветки A и изменения из ветки B переносятся в ветку A.

**git merge branch_2** - команда для слияния ветки branch_2 в текущую ветку (branch_1).

Конфликты при слиянии веток - это события, когда два изменения из двух разных веток сделаны таким образом, что при их слиянии есть вероятность потери данных в репозитории.

Merge-commit также может содержать информацию о конфликтах при слиянии и способах их решения. 
В конфликте при слиянии веток информацию об ошибках слияния можно отследить, используя команду **git status**.

Команду **git pull** нужно всегда выполнять с флагом *--rebase*, потому что при *pull* с удалённого репозитория на локальный также могут быть конфликты. 

**git rebase --continue** выполняется при разрешении конфликтов при **git pull --rebase** после добавления каждого файла в индекс после каждого решения конфликта.
